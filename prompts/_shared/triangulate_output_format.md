# Triangulate 用：出力フォーマット強制テンプレ

> あなたの出力は他の 2 つの LLM の出力と並べて diff されます。
> **以下のセクション構造を厳守**してください。逸脱すると比較不能になります。

---

## 出力テンプレ

```
# 案 [あなたの自称名] — <タスク短縮名>

## 0. 自己申告（必須・先頭に置く）

- 確信度：High / Medium / Low（このタスク全体について）
- 不安な箇所（具体的に、最低 3 つ）：
  1. ...
  2. ...
  3. ...
- 参照した文献・URL（あれば）：
- ハルシネーション可能性が高い記述（自分で気付いたもの）：

## 1. 設計判断（散文、簡潔に）

- 何を選び、何を捨てたか
- 代替案と却下理由

## 2. 疑似コード（言語非依存、まず擬似コードで書く）

- データ構造を先に定義
- 主要関数を疑似コードで
- 副作用・例外条件を明記

## 3. Python 実装（モジュール単位）

- ファイル単位で fenced code block
- ファイル名を block の先頭コメントで明示
- import 文を漏らさない

## 4. テスト（pytest 形式）

- KAT は **個別に** パラメタライズ
- 各 KAT に「この KAT が落ちたら何が間違っているか」のコメント

## 5. 自分の実装をデバッグする手順

- KAT が落ちたとき、最初に疑うべき箇所
- 中間状態のダンプ方法

## 6. 既知の限界・未実装

- やらなかったこと、後回しにしたこと
- 「これは Claude Code 側で補完してほしい」要望

## 7. 他案との差分が出そうなポイント（予想）

- このタスクで他の LLM と判断が分かれそうな設計選択を 3 つ
- 各々で「自分の選択の根拠」「他の選択肢の妥当性」
```

---

## なぜこの形式か

Claude Code は 3 案を受け取った後：

1. §0 の自己申告で「低確信箇所」をピックアップして集中レビュー
2. §2 疑似コードを 3 並列に並べて構造的差分を抽出
3. §3 Python 実装は KAT が通った部分のみ採用候補
4. §6 限界申告は人間レビューの優先順位付けに使う
5. §7 予想差分は他案との突合せのインデックスとして使う

この構造に従うほど **採用率が上がります**。
